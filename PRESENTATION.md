* **投影片要點**：
  * **背景**：AI Agent 程式碼編寫工具普及，大幅降低生物資訊分析門檻。

## 🖥️ Slide 2: 痛點剖析：AI 驅動分析時代的三大失效（Problem Statement）

* **投影片標題**：AI 驅動科學分析的三大系統性失效模式（F1–F3）
* **視覺化設計**：使用 1x3 的紅色漸層警示方塊（Red-to-Orange blocks），分別標記三類失效並附帶 icon。下方繪製一個「Token/運算成本爆炸」的連鎖流程圖。
* **簡報核心金句**：缺乏程式碼血緣記憶，科學重現性即流於空談；LLM 推理成本攀升更將此危機呈指數級放大。
* **投影片要點**：
  * **🔴 失效模式一：程式碼溯源真空 (F1)**
    * 臨時程式碼（Ad-hoc）於對話結束後消散，結果與產生它的程式碼之間無連結。
  * **🔴 失效模式二：分析方法論失效 (F2)**
    * LLM 幻覺引入錯誤的統計學標準化或過時 API，產出表面合理但污染科學結論的結果。
  * **🔴 失效模式三：分析方法漂移 (F3)**
    * 不同 Session 因微調過濾閾值或參數，引致同一原始數據產出不一致的下游結論。
* **演講備忘錄（Speaker Notes）**：
  > 傳統的 Workflow 管理器（如 Snakemake）只關心輸入檔案有沒有變。但 AI 時代的變數在於「LLM 寫出來的程式碼逻辑天天在變」。如果系統記不得上個 Session 做了什麼，就會被迫對相同意圖反覆驅動 LLM 重新生成程式碼、重新執行 L3 管道，這不僅損害可重現性，更造成了高昂的 Token 經濟開銷。

## 🖥️ Slide 3: 核心主張：程式碼血緣為可重複性之基石（Design Philosophy）

* **投影片標題**：設計哲學：以程式碼血緣為核心的 Medallion 快取架構
* **視覺化設計**：繪製一個天平，左側是「數據輸出（傳統快取）」，右側是「程式碼版本 + 執行脈絡（Evo_PRISM）」。天平明顯傾斜向右側。
* **簡報核心金句**：程式碼血緣之追溯方屬科學運算平台之根本要務；Token 節省僅為良好架構所內生之系統性質。
* **投影片要點**：
  * **數據與程式碼的關係**：數據是隨時間和環境變動的「副產物」，唯有「產生數據的程式碼版本與其 AST 正規化特徵」才是不可變的因果源頭。
  * **架構內生性質**：
    * 解決「程式碼溯源」，Token 與算力之節省即為自然推論，無需作為獨立課題處理。
    * 將「工具生命週期」與「產物依賴圖」強制內嵌於儲存層。
* **演講備忘錄（Speaker Notes）**：
  > 我們的核心哲學是「數據應為輔助，程式碼方為基石」。現有的 GPTCache 等語意快取均假設輸出為純文字，無法處理生資圖表與數據血緣。Evo_PRISM 的三層 Medallion 資料湖，從架構層面強制在資料寫入時記錄「程式碼版本 $\rightarrow$ 分析執行 $\rightarrow$ 產出產物」之完整血緣。

## 🖥️ Slide 4: 系統架構：三層 Medallion 語意資料湖（System Architecture）

* **投影片標題**：系統認知架構：L1-L2-L3 三層 Medallion 語意湖
* **視覺化設計**：展示三層堆疊的資料湖（Gold/Silver/Bronze）三色漏斗模型，並標註對應的認知記憶模型（工作記憶、情節記憶、長期記憶）。左側引用架構圖 `Figure 1`。
* **簡報核心金句**：輕量級本機資料庫 DuckDB 提供亞毫秒級查詢，實現零 Server daemon 之高效邊緣部署。
* **投影片要點**：
  * **🥇 L1 Gold（工作記憶 / 語意快取）**：
    * 本機 `bge-m3` Embedding + DuckDB HNSW 餘弦向量索引；TTL = 7 天；延遲 **< 0.001 ms**。
  * **🥈 L2 Silver（情節記憶 / 永久帳本）**：
    * `analysis_history`（永久追加）、`sample_registry`、`tools` 工具庫；延遲 **~262 ms**。
  * **🥉 L3 Bronze（長期記憶 / 唯讀原始數據）**：
    * 唯讀原始數據（Visium HD 矩陣、Kallisto 輸出）；延遲 **~34,000 ms**（冷啟動重算）。
* **演講備忘錄（Speaker Notes）**：
  > 路由閘道會攔截所有請求，並在最低成本的層次進行解析。如果是 7 天內重複問的熱點問題，由 L1 Gold 於亞毫秒級直接攔截；如果 L1 未命中，則由 L2 進行結構化 SQL 與工具版本比對；僅當兩者皆未命中時，方允許從 L3 觸發重算。這使已完成之分析在實測中達到 0 次冗餘重跑。

## 🖥️ Slide 5: 技術創新一：3-way RRF 語意快取與 Figure Cache（Contribution C1）

* **投影片標題**：零 Token 快取重用：3-way RRF 檢索與圖表剝離技術
* **視覺化設計**：左側展示 3-way RRF 公式與三路特徵（Embedding + 指紋 + 上下文），右側以對比示意圖展示「未剝離前 base64 塞爆 Context Window」與「剝離後 Figure Cache 精簡佔位符」的 Token 消耗差距。
* **簡報核心金句**：3-way 正交比對杜絕靜默快取錯誤，Figure Cache 達成 98.2% 之傳遞 Token 節省率。
* **投影片要點**：
  * **3-way RRF 排序融合 (Eq. 3)**：
    * 自然語言相似度（$w_1=1.0$）+ 輸入特徵指紋（$w_2=1.5$）+ 執行期上下文（$w_3=0.5$）。
    * 特徵指紋（檔名+大小+SHA256+schema）權重最高，防止輸入數據更新後誤中舊快取。
  * **Figure Cache 剝離技術**：
    * 將巨大之 base64 圖片內容定址寫入 `gold/figure_cache/`，LLM Context 只留 compact placeholder。
    * 單張火山圖可節省 **15-20 萬 Token**，支持長對話不爆窗。
* **演講備忘錄（Speaker Notes）**：
  > 傳統語意快取容易出現「輸入數據已變更但文字相似而誤中舊快取」的有害失效。我們設計的 3-way RRF 快取施加了最高的檔案特徵指紋權重，徹底杜絕了此類靜默快取失效。同時，結合內容定址的 Figure Cache 剝離技術，我們在 98 樣本組學管線中達成了 98.2% 的傳遞 Token 節省率，這在多模態分析中極具經濟價值。

## 🖥️ Slide 6: 技術創新二：HELIX 工具自演化框架（Contribution C2）

* **簡報核心金句**：複雜度是環境壓力，重用率是生存優勢；達爾文式選擇壓力驅動程式碼自適應演化與自癒。
* **投影片要點**：
  * **自適應晉升評估 $f_{promote}$ (Eq. 1)**：
    * 自動將「高好評、高重用、低複雜度」的動態臨時腳本晉升為 SemVer 受版本治理之正式 MCP 工具。
  * **工具健康度 HealthScore (Eq. 2)**：
    * 綜合 Churn Ratio 與複雜度增量。低於閾值 $\theta_{warning}=0.70$ 觸發 LLM 熱區自動重構，將 CC 複雜度中位數降低 **80%**，HealthScore 提升 **+0.515**。

* **演講備忘錄（Speaker Notes）**：
